# BeamDrop Mobile Screens

## Scope

This document specifies BeamDrop mobile UI for Android and iPhone. The product
model is shared, but the UI must use native conventions:

- Android: Material 3 Compose, navigation bar or rail where appropriate,
  Material sheets, snackbars, and permission education screens.
- iPhone: SwiftUI navigation stacks, lists, sheets, Share Sheet, Shortcuts
  integration, and iOS permission patterns.

Do not create a web dashboard layout on mobile.

## Global Mobile Navigation

Primary tabs or destinations:

- Home.
- Devices.
- History.
- Settings.

Secondary screens are pushed or presented:

- Pair New Device.
- Scan QR Code.
- Device Detail.
- Send Text.
- Send File.
- Transfer Progress.
- Privacy.
- Network Diagnostics.

## Onboarding

Screens:

1. Welcome: "BeamDrop" and a concise local-first promise.
2. Local transfer: explain nearby transfer without login or cloud upload.
3. Pairing: QR pairing and trusted devices.
4. Permissions: local network, camera, notifications, and files as needed.
5. Clipboard: platform-specific explanation.

Android clipboard copy: "Android limits background clipboard access. BeamDrop
sends clipboard content from user-triggered actions."

iPhone clipboard copy: "iPhone does not allow silent background clipboard
monitoring. Use Share Sheet, Shortcuts, or Paste to send clipboard content."

## Home

Content:

- Device name header.
- Local network status chip.
- Primary "Send" button.
- "Pair device" secondary action.
- Trusted device row/list.
- Recent transfer preview.

Empty state:

- Title: "No trusted devices yet".
- Body: "Pair with QR code to send locally without logging in."
- Actions: Pair with QR, Scan QR, Manual IP.

## Nearby Devices

Content:

- Scanning state.
- Trusted nearby devices.
- Unknown nearby devices.
- Manual IP action.
- Diagnostics link.

Unknown devices:

- Show as unknown, not as normal send targets.
- Actions: Pair, Ignore.
- Do not show "Send" until trust or approval rules allow it.

## Pair New Device

Content:

- QR code generated by this device.
- Expiration countdown.
- Refresh QR action.
- Scan QR action.
- Manual IP fallback.

Layout:

- iPhone: navigation screen with QR centered and actions below.
- Android: Material top app bar, QR surface, supporting text, action buttons.

## Scan QR Code

Content:

- Native camera preview.
- Framing guide.
- Permission explanation before OS prompt.
- Manual IP fallback.
- Error states for denied camera, expired QR, and invalid QR.

Accessibility:

- Provide manual pairing fallback for users who cannot use camera scanning.
- Screen reader announces scan state and permission reason.

## Device Detail

Content:

- Device name.
- Platform.
- Trust state.
- Last seen.
- Last transfer.
- Route availability.
- Send actions.
- Revoke trust.

Revoked device state:

- Disable send.
- Offer Re-pair with QR.
- Explain that revocation blocks future trusted sends and large transfer resume.

## Send Text

Content:

- Native text input.
- Character/size hint for large text.
- Device picker.
- Send action.
- Paste action where allowed.

iPhone must treat paste as user-driven. Android must avoid background clipboard
assumptions.

## Send File

Content:

- Native file picker entry.
- Selected item list.
- Total size.
- Target device.
- Send action.

Folders:

- Use platform-supported folder selection.
- Show relative path count and total size where available.

## Receive Prompt

Presentation:

- iPhone: sheet or alert depending on context and urgency.
- Android: Material dialog or full-screen activity from notification when app is
  not foreground.

Content:

- Sender name and platform.
- Trust state.
- Transfer type.
- Item count.
- Total size.
- Destination.
- Accept.
- Reject.

Unknown device note:

- "Accepting this transfer does not trust this device."

## Transfer Progress

Content:

- Sending/receiving label.
- Other device.
- Route label: Local, Manual IP, or future Relay.
- Current item.
- Progress bar.
- Bytes transferred.
- Speed.
- Time remaining when stable.
- Pause, cancel, resume, retry.
- Verification state.

Large transfer interruption must show a clear "Resume available" state.

## History

Content:

- Transfer rows grouped by date.
- Direction.
- Device.
- Content type.
- Size.
- Status.
- Actions: open, retry, resume, clear.

Privacy:

- Avoid retaining sensitive text or clipboard previews by default.

## Trusted Devices

Content:

- Trusted device list.
- Platform icons.
- Last seen.
- Revoke action.
- Pair new device.

List rows must use native swipe/context actions where appropriate, but destructive
revocation requires confirmation.

## Settings

Content:

- Device name.
- Default receive location.
- Notifications.
- Clipboard settings.
- Auto-accept trusted devices where supported.
- Network diagnostics.
- Privacy.
- About.

Mobile settings must not imply always-on clipboard sync.

## Privacy

Content:

- Local-first explanation.
- No login required for local transfer.
- Trusted devices stored locally.
- Clipboard rules.
- Clear history.
- Revoke devices.

## Network Diagnostics

Content:

- Local network permission.
- Current network.
- Discovery status.
- Manual IP.
- QR fallback.
- Firewall/client isolation explanation.

Copy must explicitly mention that public or corporate Wi-Fi may block local
discovery.
